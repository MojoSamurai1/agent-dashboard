    Code Writing Protocol

    Status: Defined here — Rollout in progress (from 2026-03-02 Audit)
    Standard defined in this section. CODE-WRITING-PROTOCOL.md to be published by docs-scribe. Schultz gate activated for all new PRs. Owner: docs-scribe + schultz.

    Golden Rules

    Rule 1: Every Function Has a One-Liner

    Every function or method must have a clear, plain-English one-liner comment explaining what it does. No jargon. Write it so a new developer or AI agent understands it in 5 seconds.

    Rule 2: Every Logical Block Has Context

    Every logical code block (if-chain, loop, DB query, API call) must have a comment explaining WHAT it does AND WHY that decision was made. Not just the what — the why matters most.

    Rule 3: No Assumed Knowledge

    Comments must be self-contained. Do not assume the reader has read external docs, ADRs, or Slack history. Every file must make sense in isolation. Reference baseline or ADR names if relevant.

    Rule 4: AI-Agent Readable

    AI agents read this code to implement features. Comments must be precise enough that an AI agent can correctly extend or modify the code without consulting external docs. If an AI gets confused, the comment failed.

    Rule 5: Docblocks for Every Public Method

    Every public PHP method needs a full PHPDoc block (@param, @return, @since, @throws). Every exported TypeScript function needs a JSDoc block. Private methods need at minimum a one-liner.

    Rule 6: Schultz Enforcement Gate

    Schultz checks code documentation before qa-lead. Missing one-liners or undocumented WHY decisions = automatic block. This is enforced, not optional.

    PHP / WordPress — Before and After

    BAD — No context, no why
    function get_order_items( $order_id ) { global $wpdb; $results = $wpdb->get_results( $wpdb->prepare( "SELECT * FROM wp_sf_order_items WHERE order_id = %d", $order_id ) ); return $results; }
    GOOD — One-liner, why explained
    /** * Fetch all line items for a given order. * * Uses a raw query instead of WP_Query because sf_order_item * is a custom table (not a CPT) -- WP_Query cannot reach it. * See baseline: core-domain-model-and-services.md * * @param int $order_id Post ID of the sf_order CPT. * @return array Raw row objects; empty array if none. * @since 0.2.0 */ function get_order_items( int $order_id ): array { global $wpdb; // Prepare prevents SQL injection -- $order_id cast to int // but prepare() is still required by our security baseline. $results = $wpdb->get_results( $wpdb->prepare( "SELECT * FROM wp_sf_order_items WHERE order_id = %d", $order_id ) ); // Return empty array (not null/false) so callers can // always safely iterate without a null check. return $results ?: []; }

    TypeScript / React — Before and After

    BAD — No context, magic numbers
    export function scoreWine(reactions: string[]): number { const weights: Record<string, number> = { love: 3, like: 2, meh: 1, dislike: 0, }; const total = reactions.reduce( (sum, r) => sum + (weights[r] ?? 0), 0 ); return Math.min(100, (total / reactions.length) * 33); }
    GOOD — Purpose and weights explained
    /** * Calculate a 0-100 preference score from emoji reactions. * * The 4-face reaction system (love/like/meh/dislike) maps to * weighted points. Weights chosen in product spec 2026-02-10 * to give 'love' 3x the influence of 'like' -- strong positives * should dominate the matching algorithm. * * @param reactions - Array of reaction strings from wine log. * @returns - Score 0-100. Higher = stronger preference. */ export function scoreWine(reactions: string[]): number { // Weight map: love=3, like=2, meh=1, dislike=0 // Decided in MyWineMojo product spec 2026-02-10. const weights: Record<string, number> = { love: 3, like: 2, meh: 1, dislike: 0, }; // Sum the weights; unknown reactions default to 0. const total = reactions.reduce( (sum, r) => sum + (weights[r] ?? 0), 0 ); // Multiply by 33 to scale 0-3 range to ~0-100, // then clamp so pure 'love' array does not exceed 100. return Math.min(100, (total / reactions.length) * 33); }

    Architecture

    Samurai Framework — Three-Layer Architecture

    Layer 1: Capability Layer

    Capabilities/ — Feature modules using singleton pattern. Each registers CPTs, taxonomies, shortcodes and admin UI. Examples: Kanban, Order, Portal, Brand, CRM, EmailIntelligence, MYOB, MarketingPlanner.

    Layer 2: Service Layer

    includes/Services/ — Business logic with static methods. Pattern: validate → check permissions → sanitize → write. Returns true|WP_Error. Never fatal. No die(), exit(), or wp_die().

    Layer 3: Domain Layer

    includes/Domain/ — Status catalogs as single source of truth. Never hardcode status values — always consume from SF_Order_Item_Status_Catalog or SF_Order_Status_Catalog.

    Bootstrap Flow

    samurai-framework.php --> plugins_loaded hook --> SF_Framework::instance() --> loads all capabilities (singleton pattern) --> registers admin menu --> enqueues assets (via wp_enqueue_script / wp_enqueue_style) --> ACF field groups sync from Data-Structures/ JSON files

    Multi-Tenancy Model

    Tenant Structure

    Organizations (sf_org CPT) are the top-level tenant. Users linked via sf_org_id user_meta. Boards, brands, and portals scoped via _sf_board_org_id post_meta. All queries filter by org ID.

    Custom Post Types

    sf_org, sf_brand, sf_brand_asset, sf_order, sf_order_item, sf_board, sf_kanban_card, sf_program, sf_marketing_calendar, sf_marketing_planner, sf_new_job_request, sf_contact (CRM)

    Key Concepts

    Master Blaster Protocol

    Every response begins with a status header: "Master Blaster 'Samurai Framework' is asking [Agent] to [action]". Provides full transparency on which agent is acting and why. Non-negotiable for all responses.

    Fail-Closed Authentication

    Assume users are NOT authenticated unless proven otherwise. Always check is_user_logged_in() before showing protected content. Default to deny. Never trust client-side data.

    Service Return Pattern

    All service methods return true|WP_Error. Never fatal errors. No die(), exit(), or wp_die() in service layer. Callers decide how to handle errors.

    Archive Not Delete

    Files are NEVER deleted. Move to .claude/archive/ with an _ARCHIVED.md companion file documenting original path, date, reason, and approval. No exceptions.

    Shortcode Registry

    Query the registry before using ANY shortcode: GET /wp-json/shortcode-registry/v1/shortcodes. Never assume a shortcode exists. Never invent attributes. See docs/shortcode-registry-QUICKREF.md.

    Canonical Precedence

    When instructions conflict: Design Lock > Architecture docs > Known-Good Baselines > AI-Human Working Rules > AI Operating Instructions > Existing code > AI suggestions.

    Feature Registry

    Consult .claude/FEATURE-REGISTRY.md before working on any feature. It maps: Branches → Roadmap → Build Plans → User Journeys → Research → Code. Always check before starting work.

    Status Catalogs

    Never hardcode status option arrays. Always consume from SF_Order_Item_Status_Catalog or SF_Order_Status_Catalog. These are the single source of truth for domain status values.

    Baseline Change Process

    To modify a FROZEN file: identify baseline by name, restate the Behaviour Guaranteed, get explicit approval. Breaking changes require marking baseline as SUPERSEDED and creating a new baseline declaration.

    Quick Start: What Are Skills?

    Agent A specialized AI role with a specific persona (e.g., "Pete" = architect, "Dave" = M365 specialist)
    Skill A reusable utility command invoked via /skill-name. NOT agent-specific — any agent or user can use it.
    Memory Persistent knowledge stored in project memory files (e.g., MEMORY.md). Agent-specific, project-scoped.
    Process Repeatable workflow sequences documented in agent prompts or CLAUDE.md (e.g., "New Feature Workflow")
    Pattern Best practices and conventions (naming, architecture, code style) enforced across projects.

    How Skills Work

    Storage: Skills live in ~/.claude/skills/<skill-name>/SKILL.md (global) or ./.claude/skills/<skill-name>/SKILL.md (project-level)

    Invocation: User types /skill-name or Claude auto-invokes based on description match

    Isolation: Optional context: fork runs skill in isolated subagent to protect main context

    Safety: Use disable-model-invocation: true for high-risk operations (deploy, git push)

    Tools: Can restrict to specific tools via allowed-tools parameter
